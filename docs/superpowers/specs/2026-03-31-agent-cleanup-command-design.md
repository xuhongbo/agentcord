# agentcord Discord 批量会话清理命令设计

日期：2026-03-31  
状态：已批准，待用户审阅  
范围：第一阶段设计

## 1. 背景

当前 `agentcord` 已支持：

- `/agent spawn` 创建主会话频道
- `/agent list` 查看当前项目下的活跃主会话
- `/agent archive` 归档当前会话到 `#history`
- `src/session-housekeeping.ts` 中的底层批量清理能力

但当某个项目分类下积累了大量历史会话时，用户只能逐个进入频道手动执行归档，操作成本高，且很难快速避开仍在运行中的会话。

本设计目标是在 `Discord` 中补充一个安全的批量清理入口，让用户可以在当前项目范围内，一次性归档“其他空闲会话”，同时保留关键频道并跳过进行中的会话。

## 2. 目标与非目标

### 2.1 目标

第一阶段目标：

1. 提供 `/agent cleanup` 命令
2. 范围限定为“当前项目分类”
3. 保留 `control`、`#history`、当前频道
4. 只处理空闲主会话，跳过进行中的会话
5. 执行前先展示预览，并要求用户显式确认
6. 执行时复用现有 `archiveSession(...)` 归档逻辑
7. 批量处理时支持部分成功、部分失败，并返回汇总结果

### 2.2 非目标

第一阶段明确不做：

- 清理整个服务器的所有项目
- 强制处理进行中的会话
- 按名称、年龄、标签等高级筛选
- 跨项目批量操作
- 将待确认请求持久化到磁盘
- 单独的管理面板或常驻控制消息

## 3. 命令定位

本功能采用：

```text
/agent cleanup
```

选择放在 `/agent` 下，而不是 `/project` 或独立面板，原因如下：

1. 该动作本质上是“批量处理代理会话”，与 `/agent archive` 同类
2. 命令入口更符合现有用户心智
3. 可以最大化复用当前会话、归档、按钮交互、项目上下文解析能力
4. 第一阶段实现成本最低、风险最小

## 4. 用户体验与交互流程

### 4.1 入口

用户可在当前项目分类下的任意频道执行 `/agent cleanup`。

要求：

- 能识别出当前项目分类
- 不要求当前频道一定是会话频道
- 在 `control` 频道中执行应是合法路径

### 4.2 预览阶段

命令执行后，机器人先扫描当前项目分类中的会话与关键保留频道，生成仅自己可见的预览消息。

预览消息至少包含：

- 项目名
- 作用范围说明
- 保留项
- 跳过中的进行中会话
- 待归档会话
- 预计处理数量摘要

并带两个按钮：

- `确认归档`
- `取消`

### 4.3 确认阶段

用户点击 `确认归档` 后：

1. 机器人校验该按钮点击人是否为原始发起人
2. 机器人读取对应的待确认请求
3. 机器人对候选会话重新做一次实时校验
4. 机器人执行批量归档
5. 机器人更新原预览消息为结果态并移除按钮

### 4.4 取消阶段

用户点击 `取消` 后：

1. 删除内存中的待确认请求
2. 把原消息更新为“已取消”
3. 移除按钮

## 5. 范围与筛选规则

命令仅处理“当前项目分类下的主会话”。

### 5.1 必须保留

以下对象绝不进入候选集：

1. 当前频道
2. 项目 `control` 频道
3. 项目 `#history` 频道

### 5.2 必须跳过

以下对象在预览中单独列出，但不会归档：

1. `isGenerating === true` 的主会话
2. 非主会话对象，例如子代理线程
3. 识别异常、无法安全判断归属的对象

### 5.3 可归档候选

只有同时满足以下条件的会话才进入归档候选：

1. 属于当前项目分类
2. 类型为持久主会话
3. 不等于当前频道
4. 不等于 `control`
5. 不等于 `#history`
6. 当前未在生成中

## 6. 执行语义

### 6.1 归档行为

每个候选会话采用现有 `archiveSession(session, guild, summary?)` 逻辑处理，保持以下行为不变：

1. 需要时自动创建或复用 `#history`
2. 在 `#history` 中创建归档帖子
3. 删除原会话频道
4. 结束会话记录
5. 写入归档存储

### 6.2 再校验原则

预览只是计划视图，真正执行时必须再校验一次当前状态。

例如：

- 某个候选会话在预览后变为进行中，则执行阶段应转为跳过
- 某个候选会话在预览后已被删除，则应记录为缺失或失败

### 6.3 部分成功原则

批量操作不采用全有或全无事务。

规则：

- 单个会话失败不影响其余会话继续执行
- 最终返回成功、跳过、失败的汇总
- 失败项需要附带会话标识与错误信息

## 7. 模块与文件设计

### 7.1 `src/commands.ts`

新增 `/agent cleanup` 子命令。

第一阶段不增加额外参数，避免命令面复杂化。

### 7.2 `src/command-handlers.ts`

新增：

- `handleAgentCleanup(interaction)`
- `handleAgent` 中的 `cleanup` 分发分支

职责：

1. 解析当前项目分类
2. 调用预览构建函数
3. 生成预览消息与确认按钮
4. 创建并登记一次性待确认请求
5. 在无候选时直接返回提示

### 7.3 `src/session-housekeeping.ts`

新增两类能力：

#### A. 预览构建

建议新增类似函数：

- `buildProjectCleanupPreview(...)`

输出至少包含：

- `categoryId`
- `projectName`
- `protectedChannels`
- `archiveCandidates`
- `skippedGenerating`
- `skippedUnknown`

该函数只负责筛选，不做实际归档。

#### B. 批量归档执行

建议新增类似函数：

- `archiveSessionsById(...)`

职责：

1. 按会话标识逐个读取会话
2. 再次检查当前状态
3. 调用 `archiveSession(...)`
4. 汇总成功、跳过、失败结果

### 7.4 `src/archive-manager.ts`

尽量不改动核心逻辑，继续复用现有 `archiveSession(...)`。

如有需要，可允许批量调用时传入更明确的归档摘要文本，但这不是第一阶段的硬要求。

### 7.5 `src/agent-cleanup-request-store.ts`

新增轻量内存请求存储模块，用于保存“预览后待确认”的批量清理请求。

建议字段：

- `requestId`
- `userId`
- `guildId`
- `categoryId`
- `currentChannelId`
- `candidateSessionIds`
- `createdAt`

行为要求：

- 仅内存存储
- 支持过期回收，例如 10 分钟
- 确认后删除
- 取消后删除
- 重复确认时视为失效

### 7.6 `src/button-handler.ts`

新增两类按钮分支：

- `cleanup:confirm:<requestId>`
- `cleanup:cancel:<requestId>`

确认分支职责：

1. 校验发起人
2. 读取请求
3. 校验请求未过期
4. 检查项目级互斥锁
5. 调用批量归档执行
6. 更新原消息为结果态并移除按钮

取消分支职责：

1. 校验发起人
2. 删除请求
3. 更新原消息为取消态并移除按钮

## 8. 互斥与并发控制

第一阶段增加一个项目级轻量互斥锁，例如：

- `activeCleanupCategoryIds: Set<string>`

规则：

1. 同一项目分类同一时间只能执行一个批量清理
2. 若某分类已在执行，则后续确认请求直接拒绝
3. 无论成功或失败，最终都必须释放锁

这样可以防止：

- 多人重复确认同一项目清理
- 同一人快速重复点击
- 不同预览请求并发操作同一批会话

## 9. 消息文案设计

### 9.1 预览消息

建议结构：

```text
批量清理预览

项目：demo
范围：当前项目分类下的其他空闲会话

将保留：
- 当前频道：<#...>
- 控制频道：<#...>
- 历史归档：<#...>

将跳过（进行中）：
- <#...> codex-fix-auth

将归档：
- <#...> codex-bug-1
- <#...> claude-docs-update

预计归档 2 个频道，跳过 1 个进行中的会话。
```

### 9.2 成功结果

```text
批量清理完成

- 已归档：2
- 跳过进行中：1
- 失败：0
- 保留：当前频道、control、history
```

### 9.3 部分失败结果

```text
批量清理完成

- 已归档：1
- 跳过进行中：1
- 失败：1

失败明细：
- <#...> codex-bug-1：缺少频道管理权限
```

### 9.4 取消与失效

取消：

```text
本次批量清理已取消。
```

失效：

```text
这次清理请求已失效，请重新执行 /agent cleanup。
```

## 10. 错误处理与边界条件

### 10.1 无项目上下文

若当前频道无法解析到项目分类，应直接返回错误提示，不进入预览。

### 10.2 无候选对象

若没有任何可归档的空闲会话，应直接提示“没有可清理的空闲会话”，不渲染确认按钮。

### 10.3 `#history` 缺失

应依赖现有归档逻辑自动创建或复用，而不是在命令层另写一套处理。

### 10.4 会话状态漂移

预览到执行之间若状态变化，应以执行阶段实时状态为准。

### 10.5 请求过期

按钮点击时若请求不存在或已过期，应提示重新执行命令。

## 11. 测试策略

### 11.1 纯逻辑测试

围绕筛选与汇总逻辑验证：

1. 当前频道不会进入候选集
2. `control` 不会进入候选集
3. `#history` 不会进入候选集
4. 进行中会话会被跳过
5. 仅处理当前分类下的持久主会话
6. 空候选时返回正确结果

### 11.2 命令处理测试

围绕 `/agent cleanup` 命令入口验证：

1. 能成功进入预览分支
2. 预览消息包含保留、跳过、候选分区
3. 无项目上下文时返回正确错误
4. 无候选时直接返回提示

### 11.3 按钮交互测试

围绕确认与取消流程验证：

1. 只有原发起人可以确认或取消
2. 请求过期时会提示失效
3. 确认后移除按钮
4. 取消后移除按钮
5. 并发确认会被互斥锁拦截
6. 部分失败时结果汇总正确

## 12. 后续扩展方向

若第一阶段稳定，可在后续再考虑：

1. 按会话年龄筛选
2. 只清理某个提供方
3. 预览中显示最近活跃时间
4. 增加“包含当前频道以外全部空闲会话”的计数提示优化
5. 提供更丰富的归档摘要来源

## 13. 结论

第一阶段采用 `/agent cleanup`，以“当前项目分类下的其他空闲会话批量归档”为目标，强调：

1. 范围明确
2. 关键频道强保护
3. 进行中会话强跳过
4. 执行前先预览、后确认
5. 底层复用现有归档能力
6. 并发安全与部分失败可观测

该方案在用户体验、实现成本、风险控制三者之间取得了较好的平衡，适合作为下一步实现计划的基础。
