# agentcord 本地会话实时感知与人工门控内建化设计

日期：2026-03-31  
状态：已批准，待规格审阅  
范围：第一阶段设计

## 1. 背景

`agentcord` 当前已经具备：

- `Discord` 发起会话与执行任务的主链路
- 常驻状态卡、摘要卡、交互卡
- `Codex` 日志增量监控
- monitor 模式工作流
- 项目挂载与会话持久化

但当前“本地终端直接发起的会话”仍然存在三个关键缺口：

1. 新会话进入 `Discord` 的速度不够快
2. 本地 `Claude` 会话缺少 `clawd-on-desk` 式的实时事件回流
3. 人工介入虽然已有基础能力，但缺少统一的双入口协调模型

本设计的目标不是桥接或依赖 `tmp/clawd-on-desk`，而是把其已经验证过的高价值能力完整搬入 `agentcord` 自身：

- `Claude` 原生钩子实时上报
- `Codex` 日志增量监控与首事件快速发现
- `Discord` 状态面板
- 人工门控闭环
- 重启后的可信恢复

## 2. 目标与非目标

### 2.1 目标

第一阶段目标是让本地 `Claude` 与 `Codex` 会话在 `agentcord` 内具备稳定的：

1. 新会话快速发现
2. 本地状态实时回流到 `Discord`
3. `Discord` 辅助人工介入
4. 终端与 `Discord` 双入口协调
5. 重启后的可信恢复

### 2.2 非目标

第一阶段明确不做：

- 继续依赖或运行 `tmp/clawd-on-desk`
- 复杂权限规则建议或策略写回
- 在 `Discord` 中回答自由文本问题
- 非受管 `Codex` 会话的稳定远程审批能力
- 项目级总览面板
- 子代理全景状态系统

## 3. 产品定位

一句话定位：

> 让 `agentcord` 在保留终端原生工作方式的前提下，把本地 `Claude` / `Codex` 会话的实时状态与可辅助人工介入能力内建到 `Discord` 作战面板。

两个硬约束：

1. 终端始终保留原生处理能力
2. `Discord` 只是辅助观察与辅助介入入口，不替代终端

## 4. 设计原则

1. 完全内建，不依赖 `tmp/clawd-on-desk` 运行时
2. 执行链与感知链分离
3. 所有状态变化统一收口到状态机
4. 会话级门控优先于消息级按钮
5. 正式信号优先于日志启发式
6. 先保证可信，再追求极限实时
7. 先补新能力，再降级旧轮询角色

## 5. 总体架构

系统保留两条主干，并用统一状态层收口。

### 5.1 主干一：现有主动执行链

这条链继续负责由 `Discord` 主动发起的任务：

`Discord 消息`  
→ `src/message-handler.ts`  
→ `src/session-executor.ts`  
→ `src/providers/*.ts`  
→ `src/output-handler.ts`  
→ `src/panel-adapter.ts`

这条链保持稳定，不做推翻式重构。

### 5.2 主干二：新增本地会话感知链

这条链负责用户在本地终端直接发起的会话：

`Claude` 钩子事件 / `Codex` 日志事件  
→ 本地感知入口  
→ 会话快速注册  
→ 事件归一  
→ 状态机  
→ `Discord` 状态面板

### 5.3 统一出口

无论事件来自主动执行链还是本地感知链，最终都统一经过：

- `src/state/event-normalizer.ts`
- `src/state/state-machine.ts`
- `src/panel-adapter.ts`

## 6. 模块边界

### 6.1 本地会话发现层

职责：

- 发现新的本地 `Claude` / `Codex` 会话
- 根据 `cwd` 归属到已挂载项目
- 创建或复用 `Discord` 会话频道
- 创建或恢复 `ThreadSession`

要求：

- 多入口发现，单一路径注册
- 注册逻辑必须幂等

### 6.2 本地状态采集层

职责：

- 采集本地会话的实时运行状态
- 不直接操作 `Discord`
- 只产出统一可归一化的原始事件

分为：

- `Claude` 钩子采集
- `Codex` 日志采集

### 6.3 统一事件桥接层

职责：

- 把本地原始事件转成平台事件
- 解析会话归属
- 推进状态机与面板更新

这一层是当前 `src/codex-monitor-bridge.ts` 的通用化升级方向。

### 6.4 人工门控协调层

职责：

- 管理当前会话的人工门控记录
- 协调终端与 `Discord` 双入口
- 保证一条门控只能被处理一次
- 负责按钮失效、过期、覆盖、恢复

### 6.5 面板展示层

继续复用现有：

- `src/discord/status-card.ts`
- `src/discord/summary-handler.ts`
- `src/discord/interaction-card.ts`
- `src/panel-adapter.ts`

## 7. 本地会话快速发现

### 7.1 目标

新的快速发现能力完全复刻 `clawd-on-desk` 的主思路：

- `Claude`：事件驱动发现
- `Codex`：日志首事件发现

当前 `src/session-sync.ts` 保留，但后续降级为补漏层，不再承担主发现职责。

### 7.2 `Claude` 快速发现

方式：

- 使用 `Claude` 原生钩子
- 在会话开始或首个关键事件时立即向本地 `agentcord` 守护进程上报

行为：

1. 收到钩子事件
2. 用 `cwd` 匹配已挂载项目
3. 如果会话未注册，立即注册
4. 首个事件继续推进状态机

### 7.3 `Codex` 快速发现

方式：

- 继续使用 `src/monitors/codex-log-monitor.ts`
- 但升级为“快速发现 + 状态监控”双职责

行为：

1. 监控器读到新 `rollout` 文件的首批有效事件
2. 提取 `session_meta.payload.cwd`
3. 若当前会话尚未注册，则立即走注册流程
4. 注册成功后继续处理当前事件

### 7.4 统一注册流程

不管来源是 `Claude` 钩子还是 `Codex` 日志，注册都必须走同一路径：

1. 解析提供方
2. 解析本地会话编号
3. 解析 `cwd`
4. 用 `cwd` 归属到已挂载项目
5. 查找是否已有频道与会话
6. 没有则创建，有则复用
7. 初始化状态卡绑定
8. 把首个事件继续送入状态机

## 8. `Claude` 钩子设计

### 8.1 原则

- 完全使用 `Claude` 原生钩子机制
- 钩子脚本尽量薄
- 业务判断全部在守护进程中完成

### 8.2 钩子脚本职责

钩子脚本只负责：

- 读取事件名
- 读取标准输入中的 `session_id` 与 `cwd`
- 组装最小 payload
- 快速发给 `agentcord`
- 快速退出

不负责：

- 项目归属
- 频道创建
- 状态机推进
- 人工门控业务逻辑

### 8.3 `Claude` 事件作用

进入系统后的作用分为三类：

1. 快速发现新会话
2. 推进实时状态
3. 触发人工门控

### 8.4 文本提问边界

第一阶段 `Discord` 只支持“允许 / 拒绝”，因此：

- 真正需要自由文本回答的问题仍由终端处理
- `Discord` 只显示“等待人工”，不承担文本回答入口

## 9. `Codex` 实时状态与增强控制

### 9.1 实时状态主来源

`Codex` 的实时事实源是本地日志文件：

- `~/.codex/sessions/.../rollout-*.jsonl`

实时链路由日志监控器承担，不再为实时状态引入第二主来源。

### 9.2 当前已具备能力

现有 `src/monitors/codex-log-monitor.ts` 已具备：

- 增量读取
- `offset` 与 `partial` 缓冲
- 失活清理
- `codex-permission` 启发式判断
- 状态映射

后续保留这些能力，不做推翻式改写。

### 9.3 主路径升级

当前问题是监控器发现状态但找不到会话时会直接丢弃事件。  
后续升级为：

1. 读到首个有效事件
2. 若会话未注册，先尝试快速注册
3. 注册成功后再推进状态

### 9.4 受管 `Codex` 启动器

为了实现真正稳定的远程“允许 / 拒绝”，后续 `Codex` 需要支持受管启动器模式。

受管 `Codex` 会话的特点：

- 通过 `agentcord` 安装的代理启动器发起
- 运行中的审批等待点可被本地守护进程感知
- `Discord` 上的允许 / 拒绝能写回当前会话
- 终端仍可直接处理

非受管 `Codex` 会话：

- 仍可快速发现
- 仍可显示状态
- 但远程审批能力不保证

## 10. 人工门控模型

### 10.1 三类门控

系统内部统一区分三类人工介入：

1. 二值审批
2. 文本提问
3. 通知态

第一阶段重点实现的是二值审批。

### 10.2 第一阶段 `Discord` 交互边界

`Discord` 上只提供：

- 允许继续
- 拒绝

不做：

- 权限规则建议
- 持久策略写回
- 文本答案输入

### 10.3 双入口协调

必须同时支持：

- 终端原生处理
- `Discord` 辅助处理

协调规则：

1. 一条门控只能成功处理一次
2. 终端先处理后，`Discord` 按钮立即失效
3. `Discord` 先处理后，终端侧后续同门控操作视为已解决
4. 守护进程退出或连接断开时，待处理门控必须进入安全终态

### 10.4 `Claude` 门控

`Claude` 钩子门控是真阻塞型门控：

- 可由 `Discord` 允许 / 拒绝
- 可由终端先处理
- 真实回执句柄只保存在内存，不持久化

### 10.5 `Codex` 门控

增强后目标：

- 受管 `Codex` 会话具备真实远程“允许 / 拒绝”
- 与 `Claude` 一样进入统一门控协调层

## 11. 状态与持久化字段设计

### 11.1 继续使用 `ThreadSession`

不新造第二套会话模型。  
新增能力全部挂在现有 `ThreadSession` 上。

### 11.2 建议新增的持久化字段

- 发现来源
- 最近本地观察快照
- 远程人工控制能力标记
- 当前人工门控记录

示意字段：

- `discoverySource`
- `lastObservedState`
- `lastObservedEventKey`
- `lastObservedAt`
- `lastObservedCwd`
- `remoteHumanControl`
- `activeHumanGate`

### 11.3 建议新增的门控记录

当前人工门控记录至少应包含：

- 唯一编号
- 提供方
- 门控类型
- 是否阻塞
- 是否支持 `Discord` 决策
- 摘要
- 关联命令
- 创建时间
- 当前状态
- 解决时间
- 解决来源
- 解决动作
- 交互卡消息编号

### 11.4 运行时句柄不落盘

以下对象只放内存，不写入持久化：

- 阻塞连接回执句柄
- 受管 `Codex` 审批控制句柄
- 定时器
- 流对象
- 进程对象

### 11.5 重启恢复规则

重启后：

- 不恢复活门控句柄
- 所有未完成门控统一转为失效态
- 旧交互卡按钮必须失效
- 最近本地观察快照可以恢复展示

## 12. 迁移顺序

### 阶段一：补齐数据模型与门控骨架

- 扩展 `ThreadSession`
- 增加统一门控记录
- 增加运行时门控注册表

### 阶段二：让 `Codex` 首事件快速发现成为主路径

- 日志监控器负责快速发现
- `session-sync` 仍保留但不再承担主发现职责

### 阶段三：接入 `Claude` 钩子快速发现与状态回流

- 新本地 `Claude` 会话快速注册
- 实时状态回流到面板

### 阶段四：实现 `Claude` 真正远程允许 / 拒绝

- `Discord` 上的二值操作真实写回本地会话

### 阶段五：实现受管 `Codex` 启动器

- 让受管 `Codex` 会话具备真实远程允许 / 拒绝

### 阶段六：把 `session-sync` 降级为补漏层

- 补漏
- 对账
- 恢复

### 阶段七：清理迁移债务

- 清理过渡逻辑
- 完善验收
- 更新文档

## 13. 风险与控制

### 13.1 `Claude` 钩子被覆盖或失效

控制：

- 安装器定期校验
- 启动时检查
- 保留补漏层

### 13.2 `Codex` 非受管会话能力不一致

控制：

- 明确区分“受管”与“非受管”能力
- 只对受管会话承诺稳定远程审批

### 13.3 双入口重复处理

控制：

- 会话级门控唯一编号
- 原子状态迁移
- 按钮与终端事件双向失效

### 13.4 重启期间悬挂门控

控制：

- 启动时统一失效
- 不恢复活回执
- 状态回落到可信状态

## 14. 成功标准

第一阶段完成后，应满足：

1. 本地新开的 `Claude` 会话可以快速出现在 `Discord`
2. 本地新开的 `Codex` 会话可以在首个有效日志事件后快速出现在 `Discord`
3. 本地状态变化能实时推进现有状态卡
4. `Claude` 的二值审批可由 `Discord` 真实允许 / 拒绝
5. 受管 `Codex` 的二值审批可由 `Discord` 真实允许 / 拒绝
6. 终端先处理后，`Discord` 卡片会立即失效
7. 重启后不会保留伪活跃门控

## 15. 结论

本设计不再把 `tmp/clawd-on-desk` 当作潜在运行组件，而是把它已经验证过的能力拆解并搬入 `agentcord` 自身：

- `Claude` 原生钩子
- `Codex` 日志增量监控
- 统一状态机
- `Discord` 状态面板
- 双入口人工门控
- 受管 `Codex` 远程审批增强

最终目标是让 `agentcord` 自己成为：

> 既能运行本地智能体，又能实时感知本地会话，并把辅助人工介入能力稳定落到 `Discord` 上的本地守护系统。
