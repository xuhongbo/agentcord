# agentcord `Discord` 实时作战面板设计

日期：2026-03-30  
状态：已批准，待规格审阅  
范围：第一阶段设计

## 1. 背景

`agentcord` 当前已经具备会话创建、会话同步、输出协调、健康检查等基础能力，但在 `Discord` 场景下，主会话频道仍然更像“聊天容器”而不是“实时作战面板”。

现阶段的核心机会点不是继续堆叠命令，而是把一个主会话频道升级为：

- 可观察：一眼看到当前会话状态
- 可理解：快速知道最近发生了什么
- 可介入：等待人工时可以直接处理
- 可协作：频道成员能共享同一个现场
- 可恢复：重启、失活、卡住后状态仍可信

本设计借鉴 `clawd-on-desk` 中最有价值的能力：统一适配层、状态机、`Codex` 增量监控、人工介入闭环、失活与恢复机制，并将其翻译到 `Discord` 产品语义中。

## 2. 目标与非目标

### 2.1 目标

第一阶段目标是让 `Claude` 与 `Codex` 主会话在 `Discord` 中具备稳定的：

1. 当前状态展示
2. 最近进展摘要
3. 等待人工闭环
4. 失活 / 卡住 / 恢复能力

### 2.2 非目标

第一阶段明确不做：

- 项目级总览面板
- 复杂权限矩阵
- 远程桥接
- 原始日志镜像到频道主流
- 子代理全景面板
- 重型安装与接入向导

## 3. 产品定位

一个频道代表一个主会话。频道内保留一条常驻状态 Embed 作为主状态面板（建议 pinned）；频道中可围观、可协作、可接管等待人工流程。

一句话定位：

> 让每个智能体主会话频道，从聊天容器升级成一个可观察、可协作、可接管的实时作战面板。

## 4. 设计原则

1. 状态优先于日志
2. 固定面板优先于消息流
3. 默认低噪音，不做底层日志镜像
4. 正式事件优先于启发式推断
5. 可信度优先于极限实时性
6. 主会话优先于子代理
7. 先到先得，结果可审计
8. 宁可状态少一点，也不能让主状态不可信

## 5. 用户体验模型

一个主会话频道内保留三类核心界面对象：

### 5.1 常驻状态卡（常驻状态 Embed，建议 pinned）

职责：表达“现在是什么状态”。这条卡片只负责状态展示，只做这一件事。

建议字段：

- 主状态
- 阶段说明
- 最近更新时间
- 当前轮次或阶段编号
- 关键计数摘要
- 风险标记

禁止放入状态卡的内容：

- 最终结果正文
- 长上下文
- 文件列表
- diff
- 命令输入输出
- 工具调用细节
- 推理细节
- 验证详情

状态卡永远不能退化成总结卡或日志卡。

### 5.2 最近进展摘要卡

职责：表达“最近发生了什么”。

摘要卡应尽量转译为面向人的低频汇报，优先回答：

- 最近完成了什么
- 当前正在做什么
- 下一步是什么
- 是否存在风险或阻塞

建议仅保留高价值动作：

- 执行类：命令、工具、检索、阶段开始/结束
- 产出类：文件改动、结果生成、阶段完成
- 协作类：子代理活动、人工介入、人工处理完成
- 风险类：异常、卡住、恢复

摘要必须做强去噪：

- 同类事件归并
- 连续重复折叠
- 时间窗口合并
- 只保留最近窗口

### 5.3 交互卡

职责：当状态需要人工介入时提供直接操作入口。

第一阶段重点场景为：`awaiting_human`。

建议最小操作集合：

- 允许继续
- 拒绝
- 稍后处理
- 本地接手 / 跳到终端（若能力稳定）

### 5.4 审计反馈

任何人工操作都需要生成轻量审计反馈，至少包含：

- 谁操作了
- 操作了哪一轮
- 操作内容是什么
- 是否生效
- 生效时间

### 5.5 本轮总结与结束总结

结果类消息必须区分两种语义：

#### 本轮总结

表示“这一轮请求处理完成”，但会话并未结束。

适用场景：

- 用户发送一条消息后，本轮 agent 已给出结果
- monitor 尚未进入终止态，但当前 worker pass 已完成
- shell / 子任务等一次性处理本轮结束

要求：

- 本轮完成后，状态卡应回落到“待命”或对应非终止态
- 本轮总结不能命名为“结束总结”
- 本轮总结允许携带完整上下文

#### 结束总结

只有在真正终止语义下才能出现，例如：

- 会话被真正结束
- 会话被归档
- 会话被阻塞并收尾
- 明确进入终止态

结束总结不能在普通一轮回复后出现。

### 5.6 长度与拆分规则

- 状态卡必须保持短小、稳定
- 摘要卡必须保持短小、聚合
- 本轮总结与结束总结不得因长度超限而直接截断
- 当结果内容超过单条 Embed 或单条消息限制时，必须自动拆成多段消息或多段 Embed 连续发送，保证上下文完整可读

## 6. 状态体系

### 6.1 三层状态模型

#### 第一层：原始事件

来自提供方或日志源，例如：

- 会话开始
- 用户提交请求
- 工具开始 / 结束
- 上下文压缩开始 / 结束
- 等待批准
- 完成
- 失败
- 会话结束

#### 第二层：统一运行状态

这是平台内部使用的主状态。

建议第一阶段支持：

- `idle`
- `thinking`
- `working`
- `awaiting_human`
- `summarizing`
- `completed`
- `error`
- `stalled`
- `offline`

#### 第三层：展示状态

给 `Discord` 顶部状态卡使用的产品文案层，例如：

- 待命
- 正在思考
- 正在执行
- 等待人工处理
- 正在整理上下文
- 本轮已完成
- 出现异常
- 疑似卡住
- 已离线

### 6.2 正式状态与推断状态

内部必须区分：

- 正式状态：由提供方明确给出
- 推断状态：由时间窗、日志模式、行为特征推断

内部状态记录建议至少保留：

- `state`
- `source`
- `confidence`
- `updatedAt`

### 6.3 状态优先级

建议优先级从高到低为：

1. `error`
2. `awaiting_human`
3. `stalled`
4. `summarizing`
5. `working`
6. `thinking`
7. `completed`
8. `idle`
9. `offline`

补充规则：

- `completed` 为短时高亮态，之后自动回落到 `idle`
- 正式事件优先级高于启发式推断
- 高优先状态可以打断低优先状态

## 7. 事件来源与状态同步架构

推荐主链路：

**提供方监控器** → **统一事件归一层** → **会话状态仓库 / 状态机** → **`Discord` 面板协调器**

### 7.1 提供方监控器

职责：

- 发现会话
- 持续观察会话事件
- 发出原始事件

第一阶段建议至少有：

- `ClaudeSessionMonitor`
- `CodexSessionMonitor`

#### `Claude`

- 正式事件优先
- 推断只做补足

#### `Codex`

重点借鉴 `clawd-on-desk` 的增量尾读策略：

- 增量读取 `JSONL`
- 为文件维护偏移量
- 处理半行残缺
- 跳过陈旧文件
- 连续重复状态去重
- 清理失活文件

### 7.2 统一事件归一层

职责：

- 把不同提供方原始信号统一为平台事件
- 统一字段命名
- 标注来源、置信度、是否推断

建议平台事件示例：

- `session_started`
- `session_idle`
- `thinking_started`
- `work_started`
- `awaiting_human`
- `human_resolved`
- `compaction_started`
- `completed`
- `errored`
- `stalled`
- `session_ended`

### 7.3 会话状态仓库 / 状态机

职责：

- 维护每个主会话当前状态
- 记录最近关键动作
- 做优先级裁决
- 做完成态回落
- 做卡住、离线、恢复判定
- 为 `Discord` 提供统一视图模型

建议主会话运行快照至少包含：

- 当前主状态
- 来源正式 / 推断信息
- 最近状态更新时间
- 最近关键动作
- 是否等待人工
- 当前交互轮次
- 是否完成
- 是否异常
- 是否失活
- 最近一次人工处理信息
- 子代理数量摘要

### 7.4 `Discord` 面板协调器

职责：

- 更新固定状态卡
- 更新摘要卡
- 挂载 / 回收交互卡
- 生成审计反馈

重要约束：

- 不直接消费原始日志
- 只消费状态仓库产出的视图模型
- 状态卡更新应由状态变化驱动，而不是原始事件直接驱动

## 8. 等待人工闭环

### 8.1 出现条件

交互卡仅在以下条件满足时出现：

- 当前状态为 `awaiting_human`
- 当前轮次尚未被处理
- 当前会话仍可操作
- 没有被后续状态覆盖

### 8.2 并发规则

- 频道可见者均可操作
- 并发规则为“先到先得”
- 后端只接受该轮次第一条有效提交
- 后续提交全部视为过期或失效

### 8.3 轮次化要求

所有等待人工按钮必须绑定唯一轮次标识，避免旧按钮误操作。

### 8.4 处理链路

1. 监控层观察到等待人工
2. 统一事件进入状态机
3. 状态切换为 `awaiting_human`
4. 面板层刷新状态卡并挂出按钮
5. 用户点击按钮
6. 后端校验当前轮次与当前状态
7. 写回结果
8. 生成审计反馈并使旧按钮失效

## 9. 失活、卡住与恢复

### 9.1 卡住判定

第一阶段建议采用简单、可解释的规则：

- 一段时间无新事件
- 当前状态不是 `idle` / `completed` / `offline`
- 且没有明确结束信号

展示文案优先使用“长时间无进展”或“疑似卡住”。

### 9.2 离线判定

当满足以下条件时，可进入 `offline`：

- 长时间无事件
- 明确会话结束
- 或系统无法继续确认其存活

### 9.3 启动恢复窗口

服务重启后不应立刻把所有会话判死。建议保留一个恢复窗口，让监控器重新接管和补回状态。

## 10. 风险与缓解

### 10.1 启发式误判

风险：

- 把正常停顿误判成等待人工
- 把长任务误判成卡住

缓解：

- 区分正式与推断
- 引入置信度
- 正式事件优先于推断
- 对中低置信推断使用更保守文案

### 10.2 并发操作冲突

风险：

- 多人同时点按钮
- 旧按钮误操作

缓解：

- 轮次化
- 先到先得
- 按钮过期校验
- 审计反馈

### 10.3 频道刷屏

风险：

- 状态更新过多
- 摘要退化为日志流

缓解：

- 多编辑固定卡片，少发新消息
- 摘要按时间窗聚合
- 只保留高价值事件

### 10.4 状态可信度下降

风险：

- 更新时间不透明
- 会话结束后状态残留
- 不同提供方表现不一致

缓解：

- 固定展示最近更新时间
- 加入陈旧 / 离线回收
- 保持统一状态语义

## 11. 第一阶段推荐落地范围

### 11.1 第一阶段做

- 统一适配层
- 统一状态机
- `Codex` 增量实时监控
- `Claude` 正式事件接入
- 常驻状态卡（Embed + 建议 pinned）
- 最近进展摘要
- 本轮总结与结束总结语义分离
- 等待人工按钮闭环
- 卡住 / 离线 / 恢复机制

### 11.2 第一阶段不做

- 项目级总览
- 复杂权限体系
- 原始日志镜像
- 远程桥接
- 子代理全景面板
- 自动安装与重型接入向导

## 12. 成功标准

第一阶段完成的产品标准应至少包括：

1. 用户进入主会话频道后，能立即看懂当前状态
2. 最近进展能在不看原始日志的情况下快速理解
3. 等待人工场景可在频道内完成最小闭环
4. 频道不会因状态同步而明显刷屏
5. `Claude` 与 `Codex` 的状态体验一致
6. 服务重启后状态不会明显乱跳
7. 会话结束、失活、卡住时状态仍然可信

## 13. 测试策略

### 13.1 单元测试

覆盖：

- 原始事件到统一事件的映射
- 状态优先级裁决
- 完成态回落
- 卡住 / 离线判定
- 轮次校验与按钮过期
- 摘要聚合与去重

### 13.2 集成测试

覆盖：

- `Claude` 主会话状态流
- `Codex` 主会话日志流
- 双提供方下状态卡一致性
- 等待人工闭环
- 服务重启后的恢复行为

### 13.3 端到端验证

覆盖：

- 真实主会话频道中状态卡持续更新
- 按钮并发点击“先到先得”行为
- 会话结束后状态合理回收
- 摘要区不会退化为消息洪流

## 14. 最终建议

最推荐的第一阶段方向是：

**频道级实时作战面板 + 等待人工闭环 + 双提供方统一状态同步**

它既能最大化借用 `clawd-on-desk` 最有价值的状态系统能力，又能严格贴合 `Discord` 多人协作场景，不会把问题过早做大。
