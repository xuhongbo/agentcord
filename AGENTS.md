请始终按照以下产品诉求与交互规范工作，这些要求优先级很高，不要偏离。

# 一、总体目标
这是一个通过 Discord 管理本地 agent 会话的系统。目标不是把本地 agent 的所有内部事件原样搬到 Discord，而是让 Discord 成为一个“低噪音、强感知、可回看”的工作界面。

核心原则：
- Discord 只展示状态、阶段性进展、最终结果
- 不展示高频底层日志
- 优先避免 Discord 限流
- 所有展示都必须围绕“让人一眼看懂当前情况”设计

# 二、输出分层规则
必须严格区分三层展示，不能混用。

## 第一层：常驻状态消息
每个主会话频道只保留一条常驻状态消息，并尽量保持 pinned。
这条消息只负责“状态展示”，只做这一件事。

允许展示：
- 当前状态：待命 / 运行中 / 等待人工 / 已完成 / 阻塞 / 异常
- Provider
- Mode
- Phase
- Iteration
- Commands 数
- File Changes 数
- Subagents 数
- 更新时间

禁止展示：
- 最终结果正文
- 长上下文
- 文件列表
- diff
- 命令输入输出
- 工具调用细节
- 推理细节
- 验证详情
- 任何长文本说明

状态卡永远不能变成总结卡或日志卡。

## 第二层：低频摘要消息
频道里只维护一条低频摘要消息，优先通过编辑而不是持续新增消息。
摘要只负责说明：
- 最近完成了什么
- 当前正在做什么
- 下一步是什么
- 是否存在风险或阻塞

摘要必须是聚合后的、人可读的内容，不要逐条输出底层事件。
同类高频事件要折叠，例如：
- 命令执行很多次时，只做聚合统计
- 文件变更多次时，只展示聚合结果与少量代表性文件
- 子代理进展只保留摘要，不逐条刷屏

## 第三层：总结消息
总结消息分两类，必须严格区分。

### 本轮总结
表示“这一轮请求处理完成”，但会话并未结束。
例如用户发一句话后，本轮 agent 给出结果，这时发“本轮总结”。

要求：
- 本轮完成后，状态卡必须回到“待命”
- 本轮总结不能叫“结束总结”
- 本轮总结正文可以较完整

### 结束总结
只有在真正结束语义下才能出现，例如：
- 会话被真正结束
- 会话被归档
- 会话被阻塞并收尾
- 明确进入终止态

结束总结不能在普通一轮回复后出现。

# 三、长度与截断规则
总结消息绝对不能因为长度过长而直接截断。
如果文本超过单条 Embed 或单条消息限制，必须自动拆分成多段消息或多段 Embed 连续发送，保证用户可以看到完整上下文。

严格要求：
- 不要用简单 truncate 截断本轮总结
- 不要用简单 truncate 截断结束总结
- 长内容必须拆段发送
- 状态卡仍然保持简短，不允许因为“完整保留结果”而把长内容塞回状态卡

# 四、默认展示风格
默认模式必须是低噪音模式：
- 不要把 command 输入输出逐条发到频道
- 不要把 tool start / tool result 逐条发到频道
- 不要把 reasoning 原文逐条发到频道
- 不要把文件变更逐条发到频道

只允许：
- 状态卡持续更新
- 摘要低频更新
- 本轮总结 / 结束总结

如果存在详细模式或 verbose 模式，也不要退回到底层日志直喷。
详细模式只表示：
- 摘要更频繁
- 摘要更细
- 但仍然必须是摘要，而不是日志流

# 五、优先级与边界
请始终遵守以下边界：

1. 状态卡只做状态展示
2. 摘要只做阶段性进展展示
3. 本轮总结只表示本轮结束
4. 结束总结只表示真正终止态
5. 总结类消息不能截断，必须自动拆段
6. 不为了“信息完整”而牺牲 Discord 的低噪音体验
7. 不为了“可见性”而把底层细节塞进状态卡
8. 所有设计都要优先考虑 Discord 限流风险

# 六、主会话与子代理的展示差异
当前阶段优先保证主会话频道体验：
- 主会话频道有常驻状态卡
- 主会话频道有低频摘要
- 主会话频道有本轮总结 / 结束总结

子代理线程当前不需要和主会话完全同级的展示复杂度，除非明确要求。
不要把主会话的状态卡逻辑粗暴复制到子代理线程。

# 七、Provider 默认行为
默认 provider 优先使用 codex，除非用户明确指定其他 provider。

# 八、工作方式要求
在做任何交互改造、消息展示改造、状态同步改造时，请始终按以下方式判断：
- 这条信息是不是应该出现在状态卡里？
- 这条信息是不是应该进摘要？
- 这条信息是不是只应该进入本轮总结或结束总结？
- 如果信息很多，是否应该聚合而不是直出？
- 如果文本很长，是否应该拆段而不是截断？

如果存在歧义，优先遵循：
- 少噪音
- 强状态感知
- 不丢最终结果
- 不污染状态卡
